Background

This report compares two sources of outcome data for CYBHI EBP/CDEP clients:

  1. discharge_data — produced by report_data_cleaning.Rmd from eInsight’s raw Discharge Report export. One row per client per assessment. Contains positive_outcomes (“Yes”/“No”), which is eInsight’s own built-in outcome flag — it is not computed by the team’s code.

  2. Cleaned paired assessment files — produced by the individual *_data_cleaning.qmd scripts (one per instrument). One row per client with a matched initial and follow-up assessment. Contains positive_outcome (“better”/“no change”/“worse”), computed by the team as a three-level variable based on first and last assessment scores.

The key question is: do these two outcome definitions agree, and where they diverge, why?

Section 1: Match rates

For each instrument, we compare the number of clients in discharge_data to the number in the cleaned paired file, and report how many match on client_system_id.

We also separate out how many discharge rows have a post-assessment recorded — clients with no post cannot appear in the paired file by definition, so the share of discharge clients with a post is the more meaningful denominator for evaluating match rates.

match_summary <- imap_dfr(paired_list, \(spec, label) {

    filter(str_detect(pre_measure_s, spec$pattern))

  tibble(

})

WarningBASIS-24 client ID mismatch

BASIS-24 is an outlier: only 4 out of 93 discharge clients match the paired file, despite both files containing roughly the same number of clients (~94 and ~99 respectively).

This is not a pairing or date issue — it is a near-complete client population mismatch between the two sources. The root cause (different extraction periods, a data entry discrepancy, or a systematic ID issue) needs to be investigated before BASIS-24 outcomes can be used in reporting.

Section 2: Score agreement

For clients that match on client_system_id, we compare pre_score/post_score in discharge_data to initial_score/follow_up_score in the paired file. Scores are near-identical when clients do match, confirming that the two sources draw from the same underlying assessments.

score_summary <- imap_dfr(paired_list, \(spec, label) {
  dis <- discharge_data |>
    filter(str_detect(pre_measure_s, spec$pattern), !is.na(post_completion_date)) |>
    distinct(client_system_id, .keep_all = TRUE) |>
    select(client_system_id, pre_score, post_score)

  paired_cols <- intersect(
    c("client_system_id", "initial_score", "follow_up_score"),
    names(spec$paired)
  )

  joined <- inner_join(dis, spec$paired |> select(all_of(paired_cols)),
                       by = "client_system_id") |>
    mutate(
      pre_diff  = pre_score  - initial_score,
      post_diff = post_score - follow_up_score
    )

  tibble(
    Instrument        = label,
    `n matched`       = nrow(joined),
    `Pre: mean diff`  = round(mean(joined$pre_diff,  na.rm = TRUE), 3),
    `Pre: % exact`    = round(100 * mean(joined$pre_diff  == 0, na.rm = TRUE)),
    `Post: mean diff` = round(mean(joined$post_diff, na.rm = TRUE), 3),
    `Post: % exact`   = round(100 * mean(joined$post_diff == 0, na.rm = TRUE))
  )
})

Section 3: positive_outcomes vs. positive_outcome

discharge_data$positive_outcomes (“Yes”/“No”) comes from eInsight’s raw export and reflects eInsight’s internal outcome logic. The paired files’ positive_outcome (“better”/“no change”/“worse”) is the team’s own calculation based on first and last assessment scores. For instruments with both variables, we cross-tabulate them.

has_po <- names(paired_list)[
  sapply(paired_list, \(s) "positive_outcome" %in% names(s$paired))
]

crosstab <- imap_dfr(paired_list[has_po], \(spec, label) {
  dis <- discharge_data |>
    filter(str_detect(pre_measure_s, spec$pattern), !is.na(post_completion_date),
           !is.na(positive_outcomes)) |>
    distinct(client_system_id, .keep_all = TRUE) |>
    select(client_system_id, positive_outcomes)

  inner_join(dis,
             spec$paired |>
               select(client_system_id, positive_outcome) |>
               mutate(paired_outcome = case_when(
                 positive_outcome ==  1 ~ "better",
                 positive_outcome ==  0 ~ "no change",
                 positive_outcome == -1 ~ "worse",
                 .default = NA_character_
               )),
             by = "client_system_id") |>
    filter(!is.na(paired_outcome)) |>
    count(positive_outcomes, paired_outcome) |>
    group_by(positive_outcomes) |>
    mutate(pct = round(100 * n / sum(n))) |>
    ungroup() |>
    mutate(instrument = label)
})

Two clear patterns emerge:

“Yes” is reliable. Across every instrument, 100% of “Yes” clients in discharge_data also score as “better” in the paired file. The two sources fully agree on positive outcomes.

“No” is not equivalent to “worse”. “No” captures “worse” and “no change” as expected, but it also includes a substantial share of clients whose scores did improve by the paired file’s definition — most notably GAD-7 (46% of “No” clients) and CPSS-5-S (58% of “No” clients). This means counting “No” as “negative outcome” in reporting will over-count poor outcomes.

Section 4: GAD-7 deep dive

For GAD-7, 46% of clients discharge labelled “No” actually improved. Yet for the sample cases examined, the discharge pre/post dates and scores matched the paired file’s initial/follow-up dates and scores exactly — both sources are using the same assessment instances.

Section 5: CPSS-5-S deep dive

For CPSS-5-S, 58% of discharge “No” clients scored as “better” in the paired file. Unlike GAD-7, the mechanism here is traceable: discharge is often using a different pre-assessment instance than the paired file.

cpss_s_scores_raw <- cpss_s_raw |>
  mutate(
    client_system_id = as.numeric(client_system_id),
    date_completed   = as.Date(date_completed),
    total_score = rowSums(across(all_of(resp_cols_cpss_s), as.numeric), na.rm = TRUE)
  ) |>
  select(client_system_id, date_completed, total_score) |>
  arrange(client_system_id, date_completed)

cpss_s_no_better_ids <- discharge_data |>
  filter(str_detect(pre_measure_s, "CPSS-5"), !is.na(post_completion_date),
         positive_outcomes == "No") |>
  distinct(client_system_id, .keep_all = TRUE) |>
  inner_join(cpss_s_paired |> select(client_system_id, positive_outcome),
             by = "client_system_id") |>
  filter(positive_outcome == 1) |>
  pull(client_system_id)

cpss_s_earliest <- cpss_s_scores_raw |>
  group_by(client_system_id) |>
  slice_min(date_completed, n = 1, with_ties = FALSE) |>
  rename(earliest_date = date_completed, earliest_score = total_score)

cpss_s_no_better_detail <- discharge_data |>
  filter(str_detect(pre_measure_s, "CPSS-5"),
         !is.na(post_completion_date),
         positive_outcomes == "No",
         client_system_id %in% cpss_s_no_better_ids) |>

In 17 of 22 cases (77%), the discharge pre-assessment date is later than the earliest assessment in the raw CPSS-5 file — on average 52 days later. The scores at these two time points often happen to be the same (because the client scored identically on both dates), but the discharge record misses the earlier improvement that the raw file captures.

This confirms that for CPSS-5-S, discharge_data’s positive_outcomes is based on a designated “Initial” assessment label in eInsight rather than the chronologically first assessment. When clients were assessed before their official “Initial” collection point, that earlier assessment is invisible to the discharge calculation.
